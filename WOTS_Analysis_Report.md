# _WOTS (WSL Dotfile Stow Tool) 技术分析报告

## 1. 核心结论：重写 vs 优化？

**建议：现阶段优先进行“架构重构式优化”，而非立即用 Rust 全面重写。**

*   **理由**：当前的性能瓶颈 80% 源于**算法设计（全量递归扫描）**和 **WSL 跨文件系统 IO 限制（/mnt/c）**，而非 Python 语言本身的性能。即使换成 Rust，如果继续频繁通过 `/mnt/c` 进行 `stat` 调用，速度依然会受限。
*   **Rust 的优势**：如果项目未来涉及大量复杂的并发处理、二进制分发或极端庞大的文件树，Rust 是首选。
*   **Python 的保留理由**：Dotfile 工具属于“高频调整逻辑”类工具，Python 的开发效率和与 Shell 命令的粘合度极高。

---

## 2. 代码分析与缺陷诊断

### 2.1 `create` 功能问题分析
目前的 `create` 命令存在以下设计与实现缺陷：
1.  **路径验证过于僵硬**：`_validate_source` 函数强制要求源文件必须在特定目录下（如 $HOME 或 /mnt/c），导致无法从临时目录（如 `/tmp`）创建配置包。
2.  **Dest 计算风险**：`_compute_dest` 在处理多个源文件时，如果源文件具有不同的层级深度，会导致目标包内的结构错乱。
3.  **破坏性移动**：对于 Linux 配置，它直接执行 `shutil.move`。如果创建过程因权限或磁盘空间失败，原配置文件可能已经丢失。
4.  **Windows 路径黑盒**：对 Windows 路径的处理依赖于字符串替换和 `/mnt/c` 的硬编码映射，在非标准挂载或非 C 盘环境下会失效。

### 2.2 `list` / `stats` 性能瓶颈
这两个命令之所以慢，是因为采用了**“深度优先全量状态同步”**策略：
1.  **重复扫描**：`count_files` 和 `dir_size` 使用 `os.walk` 遍历一次，`check_copy_status` 又遍历一次。
2.  **跨分区 IO 噩梦**：在 WSL 中，访问 `/mnt/c` 的元数据（stat）非常缓慢。`check_copy_status` 为每个文件在两端执行 `stat`。如果一个包有 1000 个文件，就会产生 2000 次耗时的系统调用。
3.  **缺乏缓存机制**：每次运行命令都要重新扫描所有物理文件，没有状态缓存或基于 mtime 的增量检查。
4.  **进程开销**：Windows 同步依赖 `pwsh.exe` 调用 `cmd /c copy`。这是一个非常重的操作，并行启动过多会导致系统卡顿。

## 3. 设计与用法报告

### 3.1 当前设计模式
*   **约定优于配置**：通过目录后缀（`.user`, `.winconfig`）自动决定同步行为。这种设计很直观，减少了额外的配置 JSON 文件。
*   **混合架构**：Linux 侧借用成熟的 `GNU Stow`（符号链接），Windows 侧使用强行覆盖（Copy）。这是一个务实的方案，避开了 Windows 不稳定的软链接支持。
*   **伪异步处理**：虽然使用了 `asyncio`，但大量底层操作仍然是 `to_thread` 包装的同步阻塞调用。

### 3.2 用法评价
*   **优点**：支持 Windows 和 Linux 统一管理，对 WSL 用户非常友好。
*   **缺点**：交互不够顺滑。`create` 命令的交互式询问（Y/n）在自动化脚本中不友好；错误信息不够结构化。

## 4. 改进建议（Roadmap）

### 4.1 性能优化（针对 Python）
1.  **合并扫描**：重构 `discover.py`，在一个 `os.scandir` 循环中同时完成：文件计数、大小计算、状态检查（mtime 比较）。
2.  **引入状态索引**：在 `.wots_cache` 中记录上次同步的 mtime 和 hash。如果文件没变，直接跳过 `stat`。
3.  **优化 Windows 同步**：
    *   **不要为每个文件启动进程**：生成一个 `powershell` 脚本块，一次性执行多个拷贝任务。
    *   **利用 `robocopy`**：它是 Windows 下最高效的同步工具，支持镜像模式和增量更新。
4.  **并发控制**：目前的 `WOTS_CONCURRENT` 默认 15 过高，跨分区 IO 下 4-8 可能更快。

### 4.2 功能增强（针对 `create`）
1.  **原子操作**：先复制到临时目录，校验成功后再删除源文件或建立链接。
2.  **支持 `--template`**：允许根据预设模板创建新的配置包结构。
3.  **智能路径映射**：允许用户通过参数指定目标路径，而非完全依赖类型推断。

### 4.3 架构升级方案
如果决定使用 **Rust** 改进：
*   **底层**：使用 `walkdir` 或 `ignore` 库进行高性能并行文件扫描。
*   **逻辑**：Rust 处理繁重的 `stat` 和路径计算，生成任务列表。
*   **调用**：通过 `tokio` 驱动的异步任务池并行处理文件拷贝。

---

## 5. 总结

`_wots` 是一个非常有潜力的工具，其“约定式目录命名”极具特色。目前的“慢”是由于对文件系统特性的处理过于简单。建议先优化扫描逻辑并引入增量检查机制，你会发现 Python 也能飞快。
