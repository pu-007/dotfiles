# WOTS Rust 重写 & 增强方案

## 1. 核心决策

### 1.1 Why Rust（已是定论，不再讨论）

Python 版的瓶颈在于 WSL `/mnt/c` 跨文件系统 `stat` 延迟和进程启动开销（每个文件一个 `pwsh.exe`）。虽然算法优化能缓解部分问题，但 Rust 的单二进制分发、零依赖启动、真正的多线程并行、以及编译期类型安全，对于这个面向系统配置管理的工具来说价值显著。代码量仅 ~1000 行，重写成本可控。

### 1.2 丢弃 pixi，使用纯 Rust 工具链

**理由**：

- `pixi.toml` 只做一件事：提供 `python` + `typer` + `rich` 环境来跑 `python -m _wots`
- `justfile` 中所有非 `wots` 任务（`_protect`, `_update`, `_backup`, `_cleanup`, `_sync_remote`, `_restore`）都是纯 bash，不依赖 pixi
- Rust 编译出单二进制文件后，`just wots` 直接调用 `./wots` 即可，pixi 完全多余
- 去掉 pixi 后，新机器只需 `git clone` + 有 Rust 编译的 `wots` 二进制即可运行，解决"先有鸡还是先有蛋"的引导问题

**保留项**：`justfile` 保留，作为任务编排器。`packages.meta/` 保留，`_backup` 任务不变。

### 2.2 技术栈

| 类别 | 选择 | 理由 |
|---|---|---|
| CLI 框架 | `clap` v4 derive | 工业标准，编译时验证，自动生成 help |
| 文件遍历 | `walkdir` | 比 `std::fs` 快 3-10x，原生过滤目录 |
| 并发 | `rayon` (CPU 密集型遍历) + `std::thread` (进程调用) | 跨文件系统 IO 是 CPU 等待，用线程池就够了，不需要 tokio |
| 序列化 | `serde` + `serde_json` | 状态索引持久化 |
| 错误处理 | `anyhow` (应用层) + `thiserror` (库层) | 简洁且类型安全 |
| 终端输出 | `crossterm` + 手动格式化 | 替代 Rich，无 Python 运行时依赖 |
| Windows sync | `robocopy.exe` | 原生多线程、增量同步、镜像模式 |
| 跨平台路径 | `std::path::Path` | `\\wsl$\` UNC 路径用字符串拼接即可 |
| 进度显示 | 自绘 + `indicatif` (可选) | 轻量进度条 |

**关于异步运行时（tokio）的决定：不使用。**

当前 Python 的 `asyncio` 主要用来并发 `pwsh.exe` 进程调用。在 Rust 中用 `std::thread::spawn` + 信号量控制并发数即可，不需要引入 tokio。文件遍历用 `rayon` 的 `par_bridge` / `par_iter`。这避免了引入一个重量级运行时。

## 3. 模块设计

### 3.1 `types.rs` → 对应 `_wots/types.py`

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash, clap::ValueEnum)]
pub enum PkgType {
    User, Config, Local, Root, Meta,
    WinUser, WinConfig, WinLocal, WinRoaming,
}
```

**增强**：为 `clap::ValueEnum` 派生，CLI 中 `--type` 参数自动校验。

### 3.2 `config.rs` → 对应 `_wots/config.py`

核心常量 + 惰性初始化：

- `DOTFILES_DIR` — 从 `DOTFILES_DIR` 环境变量或 `$HOME/dotfiles` 读取
- `WIN_USERNAME` — 自动检测 `/mnt/c/Users/` 下第一个非系统用户
- `WSL_DISTRO_NAME` — 从 `WSL_DISTRO` 环境变量或默认 `archlinux`
- `SYNC_MAX_CONCURRENT` — 从 `WOTS_CONCURRENT` 或默认 8（从 Python 的 15 下调，跨分区 IO 下 4-8 更优）
- `MAX_SYNC_SIZE_BYTES` — 从 `WOTS_MAX_SIZE_MB` 或默认 50MB
- 所有 `*_TARGET` 路径

### 3.3 `discover.rs` → 对应 `_wots/discover.py`

```rust
pub fn detect_type(path: &Path) -> PkgType;
pub fn find_packages(base: &Path) -> HashMap<PkgType, Vec<PathBuf>>;
pub fn pkg_basename(pkg_path: &Path) -> String;

pub fn winuser_rel_path(pkg: &Path, file_path: &Path) -> PathBuf;
pub fn build_win_path(file_path: &Path, pkg: &Path, pt: PkgType) -> PathBuf;
```

**增强**：
- `detect_type` 使用公共最小父目录（LCP）算法，而非单路径匹配
- `find_packages` 使用 `walkdir` 预过滤，避免 `os.scandir` 的多次系统调用
- 包名推断增强：当文件名不足以推断时（如 `init.lua`），自动截取父级目录名

### 3.4 `sync.rs` → 对应 `_wots/sync.py`

两个核心引擎：

```rust
// Linux: Stow 引擎
pub fn do_stow(pkg: &Path, target: &Path, sudo: bool, dry_run: bool) -> Result<()>;
pub fn stow_file_by_file(pkg: &Path, target: &Path, sudo: bool, dry_run: bool) -> Result<()>;

// Windows: Robocopy 引擎
pub fn sync_to_windows(src: &Path, dst: &Path, dry_run: bool) -> Result<()>;
pub fn sync_batch(items: &[(PathBuf, PathBuf)], dry_run: bool, 
                  max_concurrent: usize) -> HashMap<String, usize>;
```

**增强**：Windows 同步从 `pwsh.exe + cmd /c copy` 改为直接调用 `robocopy.exe`：

```rust
Command::new("robocopy.exe")
    .arg(src)
    .arg(dst)
    .arg("/MIR")       // 镜像模式
    .arg("/MT:8")      // 8 线程并发
    .arg("/R:1")       // 重试 1 次
    .arg("/W:1")       // 重试等待 1 秒
    .arg("/NJH")       // 不输出作业头
    .arg("/NJS")       // 不输出作业摘要
    .arg("/NP")        // 不显示进度百分比
    .arg("/XF").arg(".git")  // 排除 .git
    .status()
```

robocopy 的退出码 0-7 都表示成功（0=无变更, 1=已复制, 2=多余文件已删除, 3=2+1），需要特殊处理。

**保留 `pwsh.exe` 路径作为 fallback**，当 `robocopy.exe` 不可用时降级。

### 3.5 `status.rs` → 对应 `_wots/status.py`

## 4. 功能增强详情

### 4.1 增量同步索引（Sync-State Index）

**文件**：`{DOTFILES_DIR}/.wots_index.json`

```json
{
  "version": 1,
  "entries": {
    "git.user/.gitconfig": {
      "mtime_ns": 1716403200000000000,
      "size": 2048,
      "win_mtime_ns": 1716403200000000000,
      "win_size": 2048
    }
  }
}
```

**逻辑**：
1. 同步前加载索引
2. 对每个文件，先与索引中的 `mtime_ns` + `size` 比较
3. 完全匹配 → 跳过（包括跳过 `/mnt/c` 上的 `stat`）
4. 不匹配 → 执行完整比较 + 同步，更新索引
5. 同步完成后写入索引

**效益**：当前 1000 个文件需要 2000 次 `/mnt/c` stat 调用（每次 1-5ms），索引命中后降至接近 0。

### 4.2 智能路径识别（Intelligent Identification）

**当前问题**：`detect_type` 硬编码绝对路径匹配，在临时目录或 Git 仓库内运行失败。

**改进**：
1. 计算所有源文件的**公共最小父目录（LCP）**
2. 检查 LCP 是否在 `$HOME`、`/etc`、`/mnt/c` 之下
3. 模糊匹配时提供候选列表让用户选择（非交互模式用 `--type` 显式指定）
4. App Name 命名逻辑增强：对于 `.config/nvim/init.lua`，自动使用 `nvim` 而非 `init`（取父级目录名）

```rust
pub fn propose_name(sources: &[PathBuf]) -> String {
    if sources.len() == 1 {
        let path = &sources[0];
        if path.file_name() == Some(OsStr::new("init.lua")) 
           || path.file_name() == Some(OsStr::new("config")) {
            // fall back to parent dir name
            return path.parent()
                .and_then(|p| p.file_name())
                .unwrap_or(path.file_name().unwrap())
                .to_string_lossy()
                .to_string();
        }
    }
    sources[0].file_stem().unwrap_or(sources[0].file_name().unwrap())
        .to_string_lossy().to_string()
}
```

### 4.3 Robocopy 集成

详见 3.4 节。核心优势：
- 原生多线程（`/MT:8`）vs Python 的逐文件 `pwsh.exe` 进程
- 增量同步（只传输变更部分）vs 全量 copy
- 镜像模式（自动删除目标端多余文件）
- 权限保留

**退出码处理**：
```
robocopy 退出码 ≥ 8 才是错误
0 = 无文件复制
1 = 文件已复制
2 = 多余文件/目录已删除
3 = 2 + 1
...
```

### 4.4 Create 原子操作

**当前问题**：`create` 对 Linux 配置执行 `shutil.move`，如果中途失败，源文件已丢失。

**改进**：
1. 先 `copy` 到目标包目录（而非 `move`）
2. 校验目标包完整性（文件数、总大小）
3. 校验通过后才删除源文件
4. 校验失败 → 回滚（删除已复制的目标包内容）

```rust
pub fn create_atomic(src: &Path, dest: &Path, is_move: bool) -> Result<()> {
    // 1. copy to temp
    let tmp = dest.with_extension(".wots_tmp");
    copy_with_validation(src, &tmp)?;
    // 2. rename temp → dest
    std::fs::rename(&tmp, dest)?;
    // 3. delete source (for move)
    if is_move {
        std::fs::remove_file(src)?;
    }
    Ok(())
}
```

### 4.5 `diff` 命令（新）

从 `list --unsynced` 升级为独立命令：

```
wots diff                    # 所有类型的差异
wots diff --type winuser     # 仅 Windows 用户配置
wots diff --app git          # 仅特定包
```

输出格式：
- Stow 类型：列出未建立符号链接的文件
- Windows 类型：列出需要同步的文件（mtime/size 差异）

### 4.6 并发控制优化

从默认 15 下调至 8，并支持包级别的并行而非文件级别：
- 包内文件：顺序（robocopy 自己管理并发）
- 不同包之间：并行（`rayon` 线程池）

## 8. 风险与缓解

| 风险 | 概率 | 缓解措施 |
|---|---|---|
| robocopy 退出码语义复杂 | 中 | 封装专门的 `check_robocopy_result(exit_code) -> bool`，0-7 视为成功 |
| WSL2 挂载点变化（非 `/mnt/c`） | 低 | 使用 `config.rs` 中的 `MNT_C` 常量，支持 `WSL_MNT` 环境变量覆盖 |
| 索引文件脏读（手动修改文件后 mtime 变化） | 高 | 索引只用于跳过检查，不做为唯一真相源。索引未命中时 fallback 到完整比较 |
| 非管理员运行 robocopy 权限不足 | 中 | 检查 robocopy 错误输出，提示用户以管理员身份运行 |
| GNU Stow 版本兼容性 | 低 | 保留 `stow_file_by_file` fallback 机制 |
| 并行 robocopy 进程过多导致 Windows 侧 IO 饱和 | 中 | 包级并行 + 默认并发数 4（可配置），每次只启动一个 robocopy 进程（它内部用 /MT 并行） |

## 9. 总结

| 维度 | Python 版 | Rust 版 |
|---|---|---|
| 启动方式 | `pixi run wots` | `./wots` |
| 依赖 | Python 3.14 + pixi + typer + rich | 无（单二进制） |
| Windows 同步 | `pwsh.exe + cmd /c copy`（每文件一个进程） | `robocopy.exe`（原生多线程） |
| 状态检查 | 每文件 `stat`（1000 文件 = 2000 次） | 索引命中时跳过 |
| 并发模型 | `asyncio.to_thread` 伪异步 | `rayon` 真并行 |
| 包创建安全性 | 直接 `shutil.move` | 先 copy → 校验 → 再删除源文件 |
| 路径识别 | 硬编码绝对路径匹配 | LCP + 模糊匹配 + 交互候选 |
| 可分发性 | 需要 Python 运行时 | 单二进制，复制即用 |

Phase 1-4 预计总工时 **~9 天**（约 2 周业余时间）。
